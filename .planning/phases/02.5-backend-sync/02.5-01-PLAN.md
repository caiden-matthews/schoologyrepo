<objective>
Enable custom events to persist across devices and browsers using Supabase PostgreSQL database as the backend. This allows users to create a study event on their phone, close the app, open it on their desktop, and see the same event — breaking the single-device localStorage limitation.

**Purpose:** Gap-closure phase required before soft launch. Phase 2 (Core App) stores events only in localStorage, making them invisible on different devices. Phase 2.5 adds server-side persistence, enabling the multi-device sync users expect from a cloud app.

**Output:** Functional custom event sync across devices with Supabase backend, Netlify Function API endpoints, and updated study.html frontend logic. Custom events remain readable from localStorage for offline fallback.
</objective>

## Existing Code Patterns

From study.html (lines 1888-1970):
- Custom events are stored in localStorage via `getCustomEvents()` and `saveCustomEvents(arr)`
- Events have structure: `{ id, date, timeVal, timeDisplay, title, subject, type, note, custom: true }`
- Events are loaded and merged into the week view at render time (getMergedWeek function)
- saveCustomEvent() collects form data and pushes to localStorage array

From netlify/functions/sync.js:
- Uses shared OAuth signing via node-oauth library
- Fetches from Schoology API using OAuth 1.0a signatures
- Returns data in standard format with success/error handling

Current custom event flow (Phase 2):
1. User fills form in addEventModal
2. saveCustomEvent() pushes to localStorage
3. renderWeek() includes custom events via getMergedWeek()
4. Page refresh reloads from localStorage

Target flow (Phase 2.5):
1. User fills form in addEventModal
2. saveCustomEvent() → POST to /api/custom-events
3. Backend stores in Supabase with user_id (from OAuth token)
4. Frontend updates UI optimistically, then waits for confirmation
5. On page load: GET /api/custom-events to restore from backend
6. renderWeek() merges backend events + localStorage cache

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Netlify Function | User's OAuth token transmitted in Authorization header (HTTPS only) |
| Netlify Function → Supabase | Service key used for database operations (backend-only secret) |
| Supabase → Client DB | User can only query their own events (RLS policies enforce isolation) |
| Browser localStorage | Plaintext backup cache (no secrets stored, only custom events) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-025-01 | Spoofing | OAuth token extraction | Mitigate | Token hashed to user_id in DB; RLS ensures users can only access their own events. Token never logged. |

## Implementation Notes for Executor

### Key Design Decisions

1. **User Identification:** Schoology OAuth token is hashed to create a deterministic user_id. This allows secure database lookups without storing the actual token, and ensures the same user is identified even across sessions (same token → same hash).

2. **Optimistic UI:** Custom events save to localStorage immediately, syncing to backend asynchronously. This keeps the UI responsive even if network is slow. If sync fails, user sees a friendly error and the event is still available locally.

3. **RLS Policies:** Supabase Row Level Security (RLS) ensures a user can only select/insert/delete their own events. Even if someone gets an access token, the database enforces isolation at the table level.

4. **Offline Fallback:** If backend sync fails (network down, Supabase error, etc.), events still save to localStorage. This is a read-through cache that keeps the app usable offline.

5. **No Real-Time Sync (yet):** Phase 2.5 only adds pull-based sync (user refresh to see new data). Real-time push sync (WebSocket) is reserved for Phase 4+.
