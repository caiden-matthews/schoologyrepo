- **Files:** `study.html` (lines 1967-2110)

## Architecture Decisions

### 1. User Identification via Token Hashing
- **Decision:** Hash Schoology OAuth token to create deterministic user_id
- **Why:** Allows secure database lookups without storing raw tokens
- **Implementation:** SHA256 hash, truncated to 32 chars, unique per token
- **Benefit:** Same token → same hash across sessions, enabling consistent multi-device identification

### 2. Optimistic Frontend Sync
- **Decision:** Save to localStorage immediately, sync to backend asynchronously
- **Why:** Keeps UI responsive even on slow networks; offline fallback works automatically
- **Implementation:** POST/DELETE to backend fire-and-forget, failures logged
- **User Experience:** Event appears instantly; backend sync happens silently

### 3. Row Level Security (RLS)
- **Decision:** Use Supabase RLS policies for user isolation
- **Why:** Database enforces access control even if someone obtains an API key
- **Implementation:** SELECT/INSERT/DELETE restricted to user's own events (user_id match)
- **Benefit:** Multi-tenant safety; compromised client-side credentials can't access other users' data

### 4. Pull-Based Sync (not real-time)
- **Decision:** User refreshes page to see new data (pull model)
- **Why:** Simple, stateless, suitable for soft launch without WebSocket infrastructure
- **Implementation:** GET /api/custom-events on page load
- **Future:** Real-time sync (WebSocket) reserved for Phase 4+

### 5. Offline Fallback with Deduplication
- **Decision:** Keep localStorage as read-through cache; merge with backend on restore
- **Why:** Allows offline event creation; backend is source of truth
- **Implementation:** Deduplication by (date, title, subject); prefer backend version
- **Benefit:** Works offline; data merges correctly when connection returns

## Known Limitations & Future Work

### Phase 2.5 Limitations (Intentional for Soft Launch)
1. **Pull-based sync only:** User must refresh page to see events created on other devices
   - Real-time sync (WebSocket) reserved for Phase 4+
2. **No conflict resolution:** Last-write-wins for simultaneous edits
   - Full versioning/CRDT reserved for Phase 3+
3. **No edit endpoint:** Only create/delete; update requires delete + recreate
   - Edit functionality in Phase 3+ (low priority)
4. **No audit log:** Deletions not logged
   - Audit trail reserved for Phase 3+

### Code Quality Notes
- All error paths return user-friendly messages (not raw API errors)
- Token hashing prevents credential exposure in logs
- RLS policies prevent accidental data leaks
- CORS properly configured for cross-origin requests
- Syntax validated: custom-events.js, _oauth.js, study.html (edit applied correctly)
