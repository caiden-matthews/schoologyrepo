## Key Integration Points

**Data Flow:**
1. User connects via OAuth (Phase 1) → stores Schoology token in localStorage
2. study.html loads → automatic sync on page load (Phase 1 already does this)
3. User clicks "Sync" button → sync.js fetches from /api/v1/assignments, /api/v1/grades
4. Response updates DATA object → all tabs re-render from same source
5. Custom events stored separately in localStorage, preserved across Schoology syncs

**OAuth 1.0a Signing:**
All Schoology API calls require OAuth 1.0a signature (HMAC-SHA1):
- Consumer key (public, from Schoology Platform app registration)
- Consumer secret (private, stored in Netlify env var SCHOOLOGY_CONSUMER_SECRET)
- User token + token secret (stored in localStorage by callback.js)
- Request must be signed: Authorization header with oauth_signature

**Security Constraints:**
- Consumer secret NEVER appears in client-side code
- Tokens stored in localStorage (acceptable for session-only v1, not production-grade)
- sync.js validates school domain matches token's issuing school
- Sync errors must not expose tokens or sensitive data

**Schoology API Endpoints:**
- GET /api/v1/assignments?start=TIMESTAMP (returns user's assignments)
- GET /api/v1/grades (returns per-course grades)
- GET /api/v1/users/me (returns user profile)
- All calls signed with OAuth 1.0a

**Throttling Requirement:**
- 30-second minimum between syncs (prevents rate limit abuse)
- localStorage.lastSyncTime tracks the last successful sync
- Sync button disabled for 30s after sync, showing countdown
- User can retry immediately after error (no throttle on errors)

```javascript
async function syncData() {
  const now = Date.now();
  const lastSync = parseInt(localStorage.lastSyncTime || 0);
  
  if (now - lastSync < 30000) {
    const remaining = Math.ceil((30000 - (now - lastSync)) / 1000);
    showMessage(`Try again in ${remaining} seconds`);
    disableSyncButton(remaining);
    return;
  }
  
  showMessage('Syncing...');
  try {
    const response = await fetch('/api/sync');
    const result = await response.json();
    
    if (result.ok) {
      DATA = result.data;
      localStorage.lastSyncTime = result.lastSync;
      showMessage('Synced successfully');
      rerenderAllTabs();
    } else {
      showError(result.error); // e.g., "Schoology unavailable"
      // Don't update lastSyncTime on error, so user can retry immediately
    }
  } catch (err) {
    showError('Network error. Try again.');
  }
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description | Threat |
|----------|-------------|--------|
| Client ↔ Schoology API | Untrusted network; OAuth tokens transmitted | Token interception, MITM, replay attack |
| Client ↔ Netlify Functions | Browser → serverless; tokens in request headers | Token exposure in logs, function side-channel |
| localStorage | Local storage on client device | Token theft if device compromised, XSS exfiltration |
| Schoology OAuth Server | External OAuth provider | Account takeover if OAuth flow compromised |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | Schoology API responses | Mitigate | Verify HTTPS only (no HTTP); validate response structure; reject malformed JSON |
| T-02-02 | Tampering | localStorage tokens | Mitigate | Use secure HttpOnly cookies (if possible); note: v1 uses localStorage, acceptable for session-only; phase 3+ should upgrade to HttpOnly |
| T-02-03 | Repudiation | API calls | Mitigate | Log sync requests with timestamp and result (ok/error) for audit; do not log tokens |
| T-02-04 | Information Disclosure | Error messages | Mitigate | Do not expose Schoology API details, token values, or stack traces in error messages (per SEC-05) |
| T-02-05 | Information Disclosure | sync.js logs | Mitigate | Strip tokens from all console.log/error output; use regex to redact oauth_token=*** |
| T-02-06 | Denial of Service | Sync endpoint | Mitigate | Rate limit per user (30s throttle); reject > 10 syncs/minute from same IP; return 429 Too Many Requests |
| T-02-07 | Elevation of Privilege | Cross-user data access | Mitigate | Verify Schoology API tokens are scoped to requesting user (API does this); validate domain matches token's school at sync time |

**Security Summary for Phase 2:**
- Tokens are stored in localStorage (acceptable for v1 session-only; upgrade in phase 3+)
- All Schoology API calls use HTTPS + OAuth 1.0a signature (tamper-proof)
- Error messages do not expose tokens or sensitive data
- Domain validation prevents token misuse across schools
- Sync throttling (30s) prevents rate limit abuse and DoS
- No server-side user database means no data breach risk (tokens only in browser localStorage)
