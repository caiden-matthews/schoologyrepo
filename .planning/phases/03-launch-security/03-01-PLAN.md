In error handling code path:
1. Catch error scenarios: missing oauth_verifier, invalid state, HTTP 401/403/404 from Schoology API, network timeout, etc.
2. Map error code/status to errorMessages key
3. Log technical error details server-side (console.error) for debugging
4. Redirect with user-friendly message: `/#oauth_error=${encodeURIComponent(friendlyMessage)}`
5. Do NOT include error codes, stack traces, tokens, or API details in the user-facing message

Example pattern:
```javascript
if (!oauth_verifier) {
  const reason = event.queryStringParameters?.error || 'unknown';
  const message = errorMessages[reason] || 'An error occurred. Try again.';
  console.error(`[callback] OAuth denied: ${reason}`); // Log details
  return redirect(`/#oauth_error=${encodeURIComponent(message)}`); // Show friendly msg
}

// Try to exchange verifier for token
try {
  const response = await getAccessToken(oauth_verifier, oauth_secret, ...);
  if (response.status === 401) {
    console.error('[callback] 401 from Schoology');
    return redirect(`/#oauth_error=${encodeURIComponent(errorMessages['401'])}`);
  }
  // ... handle 403, 404 similarly
} catch (error) {
  console.error('[callback] network error:', error);
  return redirect(`/#oauth_error=${encodeURIComponent(errorMessages['network_error'])}`);
}
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client (browser) ↔ Netlify Functions | OAuth flow: client receives tokens, passes verifier to callback function; function returns access token. Tokens must not be logged/exposed. |
| Netlify Functions ↔ Schoology API | Server-to-server: uses OAuth 1.0a signatures (HMAC-SHA1) to authenticate. Consumer secret stored securely in Netlify env vars. |
| Client ↔ External Scripts (AdSense) | CSP controls which external scripts can load; AdSense has broad image/analytics access. CSP restricts to approved domains. |
| User Input (domain field) ↔ auth.js Function | Untrusted: user-supplied school domain must be validated before use in API endpoint construction. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | OAuth callback URL | mitigate | Callback function validates state parameter (existing); Phase 3 adds friendly error messages to prevent user confusion. |
| T-03-02 | Spoofing | Client impersonation via stolen token | mitigate | Tokens stored in localStorage (not httpOnly yet — Phase 3 uses plain storage per Phase 2 design); Phase 4 can upgrade to httpOnly cookie if needed. |
| T-03-03 | Information Disclosure | Schoology API tokens exposed in error messages | mitigate | Per D-20, no token/API details shown to user; friendly error messages only. Technical details logged server-side (console.error in Netlify functions). |
| T-03-04 | Information Disclosure | CSP bypass via XSS | mitigate | CSP header blocks unauthorized scripts via script-src 'self' + explicit domains. Phase 4 nonce-based CSP further hardens. |
| T-03-05 | Denial of Service | Sync function called repeatedly | mitigate | Existing Phase 2 implementation: sync button disabled for 30 seconds after each sync (throttling). Netlify function timeout: 10 seconds default. |
| T-03-06 | Elevation of Privilege | User accesses other user's data | mitigate | OAuth tokens are per-user; Schoology API enforces user-scoped data access. App has no server-side user database (stateless). |
| T-03-07 | Tampering | COPPA violation (ads to minors) | mitigate | Phase 3: integrate AdSense placeholder script (no real ads yet). Phase 4 task: tag site as child-directed in AdSense settings; disable interest-based ads. Per RESEARCH.md, compliance is required by April 22, 2026 (passed); Phase 4 MUST configure before revenue. **Documented in COPPA-CHECKLIST.md as Phase 4 blocking gate.** |
| T-03-08 | Denial of Service | Landing page unavailable | mitigate | Deployed on Netlify CDN (99.99% uptime SLA). Phase 4 task: set up uptime monitoring (Uptime Robot) to alert if down. |
| T-03-09 | Tampering | ads.txt domain hijacking | mitigate | ads.txt served by Netlify; signed with HTTPS. AdSense verifies file content matches publisher ID. User must safeguard Netlify account credentials. |
| T-03-10 | Information Disclosure | User school domain logged in errors | mitigate | Per D-19, domain logged server-side for debugging (in console.error); friendly error message shown to user without domain details. Consistent across auth.js, callback.js. |

**Accepted Risks:**
- T-03-02 (localStorage vs httpOnly): Acceptable for v1 (no server-side accounts); Phase 4+ can upgrade to httpOnly if needed.
- T-03-07 (COPPA Phase 4 dependency): Acceptable because Phase 3 has no revenue; Phase 4 MUST complete COPPA configuration before monetization. **CRITICAL: This is a legal/regulatory blocker, not optional.**

**Assessment:** Phase 3 CSP + friendly error handling reduce XSS, spoofing, and information disclosure risks significantly. COPPA and httpOnly cookie upgrades deferred to Phase 4 (acceptable trade-off for launch timeline, but compliance is mandatory before revenue).
