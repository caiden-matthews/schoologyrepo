<objective>
Enable any Schoology student to connect via Platform-level OAuth and see their real data immediately.

**Purpose:** Phase 1 breaks the single-school barrier. Previously, OAuth worked only for shschools.org with hardcoded consumer credentials. Now, students from any Schoology-powered school can:
1. Visit the landing page
2. Enter their school name (e.g., "myschool")
3. Authenticate via Schoology OAuth
4. See their real assignments, grades, and data

This establishes the foundation for all subsequent phases. Without working multi-school OAuth, the app remains a single-user personal tool.

**Output:** Deployable `index.html` landing page + updated Netlify functions that support dynamic school domains + working session-based auth + data isolation per browser session.

**Success Threshold:** A student from any real Schoology school can enter their domain, click "Connect," authorize, and immediately see their real data in the app (Today view, Week view, etc.). No crashes. Tokens persist across page refresh.
</objective>

**Key Implementation Patterns:**

From existing code (`netlify/functions/auth.js`, `callback.js`, `sync.js`):
- OAuth 1.0a signing is already implemented (reusable `oauthSign`, `oauthHeader` helpers)
- Request token secret stored in HttpOnly cookie (good pattern to maintain)
- Access token + secret passed via URL hash from callback to browser
- Browser picks up tokens from hash, stores in localStorage, cleans URL
- sync.js already calls Schoology API endpoints (`/assignments`, `/grades`, `/users/me`)

Current issue: All functions hardcoded for `shschools.org` OAuth flow. Phase 1 makes these generic.

**Domain parameterization strategy:**
- User enters school name only (e.g., "myschool")
- App auto-appends `.schoology.com` to form `myschool.schoology.com`
- Domain passed through OAuth state parameter (callback.js) and into sync.js
- Each API call uses the stored domain to call the correct school's endpoint

**Token lifecycle (localStorage):**
- After OAuth callback, `sc_at` (access token) + `sc_ats` (token secret) stored in localStorage
- Schoology tokens naturally expire (typically 30+ days; actual lifetime TBD by research)
- On page load, check if tokens exist; if yes, proceed to app; if no, show landing page
- No automatic token refresh in Phase 1; deferred to Phase 2 (D-09 deferred)

**Error handling strategy:**
- OAuth request fails → show friendly error at auth.js
- OAuth authorization denied → callback.js catches, redirects to app with `oauth_error` hash param
- Domain not found → sync.js detects 404/401 from Schoology API, shows user-friendly message
- Network error during sync → show "Failed to sync. Check connection and try again."

**UI Strategy:**
- Landing page: Minimal, fast. Headline + domain input + big "Connect" button. No lengthy explanation (D-01).
- Post-auth: Hide landing page, show app tabs (Today, Week, Plan, Grades, Flashcards).
- Domain input hidden once authenticated (D-07).
- Post-auth visibility enforced by checking `localStorage.getItem('sc_at')` on page load.

**Security notes:**
- Consumer key/secret NEVER in client code — Netlify env vars only (SCHOOLOGY_CONSUMER_KEY, SCHOOLOGY_CONSUMER_SECRET)
- OAuth callback hash cleaned before any scripts parse it (Phase 3, AUTH-05 compliance)
- Tokens never logged in console or error messages
- Domain validation: attempt OAuth; catch domain errors at callback time (D-03, not client-side validation)

<threat_model>
## Trust Boundaries

| Boundary | Description | Risk |
|----------|-------------|------|
| Client ↔ Schoology API | User's access token crosses from browser → Netlify function → Schoology | Token exposure if leaked |
| User Input → Backend | School domain input from landing page to auth.js | Injection attacks (domain parameter poisoning) |
| Browser ↔ Netlify Functions | Tokens passed via URL hash and localStorage | XSS if CSP not enforced (Phase 3 responsibility) |
| OAuth Callback | User redirected back from Schoology with verifier | Token verification, replay attack risk |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-S-01 | Spoofing | School domain input | Mitigate | Validate domain format in auth.js; attempt OAuth; if domain doesn't exist, Schoology will reject. No client-side blocklist needed (D-03). |
| T-01-T-01 | Tampering | OAuth state parameter | Mitigate | Use state parameter to pass school domain; state is included in OAuth signature, cannot be modified. If state is tampered, OAuth signature fails. |
| T-01-R-01 | Repudiation | Token storage in localStorage | Accept | localStorage is per-browser. No audit trail required for Phase 1 (session-only, no accounts). |
| T-01-I-01 | Information Disclosure | Access token in URL hash | Mitigate | Hash fragments never sent to server, only read client-side. Tokens stored in localStorage (not cookies) to avoid server transmission. Phase 3 adds CSP to prevent XSS. Clean hash from URL immediately after parsing (AUTH-05). |
| T-01-I-02 | Information Disclosure | Tokens in console errors | Mitigate | Do NOT log tokens in error messages. Catch errors before logging, sanitize messages. Return user-friendly errors, not raw API responses. |
| T-01-D-01 | Denial of Service | Rate limiting on Schoology API calls | Mitigate | Phase 2 implements sync throttling (30s minimum between syncs, DATA-09). Phase 1: trust user is honest. If Schoology is rate-limited, they return 429; sync.js returns error to user. |
| T-01-D-02 | Denial of Service | Invalid school domain spam | Accept | Invalid domains fail at OAuth time (Schoology rejects); no server-side cost to reject invalid domains. If attacker spams, Schoology rate-limits them, not us. |
| T-01-E-01 | Elevation of Privilege | Cross-user data leakage | Mitigate | Each user's tokens are unique; Schoology API validates and returns only that user's data. No shared state. localStorage is per-browser-session. Not possible for student A to see student B's data without student B's Schoology credentials. |

**Notes:**
- Phase 1 focuses on getting OAuth working. Security headers (CSP, HSTS) are Phase 3 (AUTH-05, SEC-01, SEC-02).
- Token refresh is deferred to Phase 2 (D-09).
- Rate limiting on client requests is Phase 2 (DATA-09).
- All Schoology API calls are signed with consumer secret (stored server-side in Netlify env vars), never exposed to client.
</threat_model>
